# 將 Claude Code 配置從 CLAUDE.md 升級到 Hooks、Skills 和自訂 Agents

> by David Chu（前 ENS 團隊）｜台北 Claude 社群聚會分享

---

## 背景

幾週前在台北 Claude 社群聚會上，軟體工程師 David Chu（前 ENS 團隊成員）在中研院為工程師們進行了一場 demo 與分享。

當時他已經同時跑著 3–5 個 Claude Code 實例，100% 的程式碼由 Claude Code 生成，每天合併 2–4 個 PR。這套流程運作得相當不錯，但他每天下班時都精神疲憊，因為得不斷切換終端機、審查輸出、重新下提示。

這些挫折讓他開始探索與 Claude Code 協作的不同哲學。

---

## 關鍵洞察：Boris Cherny 的做法

靈感來自 Claude Code 創作者 Boris Cherny 的一篇貼文——他每天能合併 10–30 個 PR，一個月內完成 259 個 PR，每一行程式碼都由 Claude 撰寫。他形容這套方式「出乎意料地平凡」，並建議大家毫不留情地投資在 CLAUDE.md 上。

> *「每次 Claude 做錯了什麼，就把它加進 CLAUDE.md，讓它永遠不再重蹈覆轍。」*
> — Boris Cherny

他自己的 CLAUDE.md 大約有 2,500 個 token，在 code review 過程中由整個團隊持續疊代。

### 兩種哲學的對比

| 哲學 | 做法 | 風險 |
|------|------|------|
| Boris 流派 | CLAUDE.md 越豐富越好，持續累積規則 | Context window 被佔滿 |
| RPI 學派 | CLAUDE.md 控制在 60 行以內，只保留普遍規則 | 過早優化 |

> Context window 被佔用超過約 40% 後效能會下降，通常稱為「**笨蛋區**」。

**David 的結論：兩種都對，取決於你所處的階段。** 先從 Boris 的方式開始，培養習慣；等規則逐漸成熟，再將它們從 CLAUDE.md 「畢業」出去。讓 Boris 那份密集 CLAUDE.md 得以運作的秘密武器，是一個強制驗證 CLAUDE.md 本身的驗證器，現已成為 `/simplify` 指令的一部分。

---

## AI 程式碼代理的五個進化層級

| 層級 | 名稱 | 說明 |
|------|------|------|
| 第一層 | 對話式 | 你輸入提示，Claude 回應，你作出反應 |
| 第二層 | 配置式 | CLAUDE.md 引導行為，記憶保留學習成果 |
| 第三層 | 自動化 | Hooks 強制執行規則，Skills 將工作流程編碼 |
| 第四層 | 編排式 | 具備特定角色、工具與記憶的自訂 Agents |
| 第五層 | 自主式 | Agent 團隊獨立運作並相互驗證成果 |

> 大多數人停留在第一或第二層。這場分享的目的是幫助大家進階到第三或第四層。

---

## 畢業路徑一：CLAUDE.md → Hook

Hooks 是在 Claude 工作流程的特定時刻自動觸發的指令。

### 常用 Hook 類型

- **PreToolUse** — 工具執行前觸發，類似非同步的條件判斷
  - 範例：防止任何對 `.env` 檔案的更動
- **PostToolUse** — 工具執行後觸發，類似非同步的回呼函式
  - 範例：自動執行格式化工具、linter、logger
- **Stop** — Claude 嘗試完成任務時觸發，作為完成前的品質關卡
  - 範例：強制 Claude 修復所有失敗的測試才能停止

### 為什麼驗證器很重要

沒有驗證器的情況下，連鎖步驟會不斷累積錯誤：

> 5 個各有 80% 可靠度的步驟 → 整體成功率只有 **33%**

加上 Stop hook 驗證器後：

> 即使單步可靠度只有 70%，只需 4 次嘗試就能達到 **99% 的整體成功率**

**核心洞察：好的驗證器 + 差的工作流程，勝過好的工作流程但沒有驗證器。**

## 畢業路徑二：CLAUDE.md → Skill

Skills 適用於模型未經訓練的知識、特殊工作流程，或是你希望以特定方式完成的任務。與其把所有內容塞進 CLAUDE.md 消耗 context，Skill 只在相關時才被載入。

- 適用於**不常使用的工作流程**（例如：涵蓋 Dexie.js 設定、模式與錯誤處理的 skill）
- 可以包含**動態內容**（例如：使用 `!` 前綴執行 bash 指令的 commit-push-pr skill）
- 可以納入**團隊偏好**（例如：說明 UI 元件庫使用偏好的章節）

---

## 畢業路徑三：CLAUDE.md → 自訂 Agent

自訂 Agents 適用於**專屬角色**，而非特定任務。

| | Skill | Agent |
|---|---|---|
| 定義 | 任務 | 人設 |
| 最適合 | 單一可重複的工作流程 | 跨多種任務執行相同角色 |

---

## 何時將規則從 CLAUDE.md 移出？

David 觀察 3 個訊號：

1. CLAUDE.md 達到一定規模（建議約 50 行）
2. Claude 不再持續遵守某條規則
3. 到了定期審查的時間點

---

## 總結

1. 當 Claude 犯錯時，**先將規則加進 CLAUDE.md**
2. 當 Claude 不再遵守規則時，**將它升級為 Hook、Skill 或 Agent**
3. 這樣能確保你的工具從**實際遇到的摩擦**中成長，而非過早優化

**nano**: 好問題！讓我解釋為什麼需要這個步驟 👇

---

*核心問題：電腦看不懂文字，只懂數字*

要做「語意搜尋」，就必須把文字轉換成數字向量，這樣電腦才能比較「哪兩段文字意思相近」。

這個轉換工作就是由 *Embedding Model（嵌入模型）* 來做。

---

*為什麼選 all-MiniLM-L6-v2？*

這個模型負責把一段文字變成 *384 個數字組成的向量*，例如：

```
"What is RAG?"
→ [0.23, -0.41, 0.87, ..., 0.12]  (384維)
```

語意相近的句子，向量也會很接近：
• "What is RAG?" ≈ "Explain RAG to me"
• 但遠離 "What is the weather today?"

---

*為什麼要「載入」它？*

它是一個訓練好的神經網路（80MB），需要先載入記憶體才能使用，就像 Word 要先開啟才能打字一樣。

載入後用途有兩個：
• *建索引時* — 把 18 篇文件全部轉成向量存到 ChromaDB
• *查詢時* — 把你的問題也轉成向量，才能比對

---

*簡單比喻*

> 想像圖書館把每本書的「主題」翻譯成同一種語言（數字），之後找書時只要看哪個數字最接近就好 — 不用一字一字對比文字。

這就是為什麼語意搜尋比關鍵字搜尋強：即使你用不同詞彙提問，只要意思一樣，還是能找到正確文件。

*為什麼要同時用關聯式資料庫？*

ChromaDB 只適合存向量和少量 metadata，但有些資訊它管理起來很不方便：

| 需求 | ChromaDB | 關聯式DB |
|------|----------|---------|
| 語意搜尋 | ✅ | ❌ |
| 儲存完整原文 | 勉強 | ✅ |
| 複雜查詢/統計 | ❌ | ✅ |
| 關聯其他資料 | ❌ | ✅ |
| 更新/刪除方便 | 麻煩 | ✅ |

---

*典型的雙資料庫架構：*

```
PostgreSQL（主資料庫）
├── documents 表
│   ├── id
│   ├── title
│   ├── content     ← 完整原文
│   ├── category
│   ├── author
│   ├── created_at
│   └── updated_at
└── ...其他業務資料表

ChromaDB（向量資料庫）
└── 只存：id + embedding
    （id 對應 PostgreSQL 的 id）
```

---

*查詢流程：*

```
使用者問題
    ↓
ChromaDB 找出最相關的 top-K 個 id
    ↓
用這些 id 去 PostgreSQL 撈完整資料
    ↓
組成 prompt 送給 LLM
```

---

*實務上常見組合：*

• *PostgreSQL + pgvector* — 直接在 PG 裡做向量搜尋，不需要獨立 ChromaDB（中小型專案首選）
• *PostgreSQL + Pinecone/Qdrant* — 大規模生產環境
• *SQLite + ChromaDB* — 輕量 demo/原型（就是現在這個系統）

---

簡單說：*ChromaDB 負責「找到誰」，PostgreSQL 負責「管理誰」*，兩個各司其職。

**nano**: RAG 有幾個主要深入方向，看你的目標選擇：

---

*🔬 方向一：提升檢索品質*

這是影響 RAG 效果最大的環節

• *Hybrid Search* — 結合向量搜尋 + 關鍵字搜尋（BM25）
• *Reranking* — 用 Cross-encoder 對檢索結果重新排序
• *HyDE* — 先讓 LLM 生成一個假設答案，再用它去檢索
• *Multi-query* — 把一個問題改寫成多個角度再分別檢索

---

*📄 方向二：更好的文件處理*

• *Semantic Chunking* — 依語意切割，不是固定字數
• *Parent-Child Chunking* — 小塊用來檢索，大塊用來生成
• *Document Hierarchy* — 保留文件的章節結構
• *Multi-modal RAG* — 支援圖片、表格、PDF

---

*🤖 方向三：更聰明的生成策略*

• *Self-RAG* — 讓 LLM 自己決定什麼時候需要檢索
• *FLARE* — 生成過程中動態判斷要不要再去檢索
• *Iterative RAG* — 多輪檢索，每次結果餵給下一輪

---

*📊 方向四：評估系統*

很多人跳過這塊，但其實很重要

• *RAGAS* — 自動評估 faithfulness、relevance、recall
• *TruLens* — 監控 RAG pipeline 每個環節的品質
• *LangSmith* — 追蹤每次查詢的完整 trace

---

*🏗️方向五：框架與工具*

• *LangChain / LlamaIndex* — 現成的 RAG pipeline 框架
• *pgvector* — 直接在 PostgreSQL 做向量搜尋
• *Qdrant / Weaviate* — 生產級向量資料庫

---

*建議學習路線：*

如果是剛開始 👉 先把 *Hybrid Search + Reranking* 搞懂，效果提升最明顯

如果要做產品 👉 加入 *RAGAS 評估* + *PostgreSQL 雙資料庫架構*

如果追前沿研究 👉 看 *Self-RAG、FLARE、Adaptive RAG* 相關論文

將 Claude Code 配置從 CLAUDE. md 升級到 Hooks、Skills 和自訂 Agents｜by David Chu（前 ENS 團隊）

–

幾週前在台北 Claude 社群聚會上，軟體工程師 David Chu（前 ENS 團隊成員）在中研院為工程師們進行了一場 demo 與分享。

當時他已經同時跑著 3-5 個 Claude Code 實例，100% 的程式碼由 Claude Code 生成，每天合併 2-4 個 PR。從許多角度來看，這套流程已經相當不錯，但他每天下班時都精神疲憊，因為他得不斷切換終端機、審查輸出、重新下提示。

這些挫折讓他開始探索與 Claude Code 協作的不同哲學。

關鍵的洞察來自 Claude Code 的創作者 Boris Cherny 的一篇貼文——他每天能合併 10-30 個 PR，一個月內完成了 259 個 PR，每一行程式碼都由 Claude 撰寫。他形容這套方式「出乎意料地平凡」，並建議大家毫不留情地投資在 CLAUDE. md 上。

「每次 Claude 做錯了什麼，就把它加進 CLAUDE. md，讓它永遠不再重蹈覆轍。」—— Boris Cherny

他自己的 CLAUDE. md 大約有 2,500 個 token，在 code review 過程中由整個團隊持續疊代。

這與 RPI 學派的觀點形成對比——後者建議將 CLAUDE. md 控制在 60 行以內，只保留普遍適用的規則。一旦 context window 被佔用超過約 40%，效能就會下降，這通常被稱為「笨蛋區」。

那麼哪種哲學才是對的？David 認為兩者都對，取決於你所處的階段。過早優化本身也可能是一個陷阱。先從 Boris 的方式開始，培養習慣，再隨著規則逐漸成熟，將它們從 CLAUDE. md 中「畢業」出去。事實上，讓 Boris 那份密集的 CLAUDE. md 得以運作的秘密武器，是一個強制執行 Claude. md 本身的驗證器。那個驗證器現在已成為 /simplify 指令的一部分。

–

David 提出了 AI 程式碼代理的五個進化層級：

→ 第一層 對話式：你輸入提示，Claude 回應，你作出反應 
→ 第二層 配置式：CLAUDE. md 引導行為，記憶保留學習成果 
→ 第三層 自動化：hooks 強制執行規則，skills 將工作流程編碼 
→ 第四層 編排式：具備特定角色、工具與記憶的自訂 agents 
→ 第五層 自主式：agent 團隊獨立運作並相互驗證成果

大多數人停留在第一或第二層。這場分享的目的，就是提供一個指引，幫助大家進階到第三或第四層。

–

畢業路徑一：CLAUDE. md → Hook

Hooks 是在 Claude 工作流程的特定時刻自動觸發的指令。幾個值得注意的 hooks 包括：

→ PreToolUse — 在工具執行前觸發，作用類似非同步程式設計中的條件判斷。適合用來防止特定檔案被讀取或修改，例如防止任何對 .env 檔案的更動。
→ PostToolUse — 在工具執行後觸發，作用類似非同步程式設計中的回呼函式。適合用來執行格式化工具、linter 或 logger。
→ Stop — 在 Claude 嘗試完成任務時觸發，作為完成前的品質關卡。適合加入驗證器，讓 Claude 在通過驗證前無法停止。例如強制 Claude 修復所有失敗的測試，即使 Claude 認為這些測試超出範疇。

附帶驗證器的 Stop hooks 非常重要。沒有驗證器的情況下，連鎖步驟會不斷累積錯誤——五個各有 80% 可靠度的步驟，整體成功率只有 33%。但加上 Stop hook 的驗證器後，即使單步可靠度只有 70%，只需 4 次嘗試就能達到 99% 的整體成功率。David 的核心洞察：好的驗證器搭配差的工作流程，勝過好的工作流程但沒有驗證器。
